# Software Requirements Specification (SRS)

## TeXPulse Studio

**Document version:** 1.0  
**Status:** Draft for implementation  
**Date:** 13 June 2026  
**Provisional product name:** TeXPulse Studio  
**Primary target platform:** Windows 11  
**Initial release model:** Offline, single-user desktop application  

---

## 1. Document purpose

This Software Requirements Specification defines the requirements, architecture boundaries, delivery phases, sprint plan, testing strategy, acceptance criteria, and quality gates for **TeXPulse Studio**.

TeXPulse Studio is an offline desktop LaTeX development environment that provides an Overleaf-like editing experience while compiling documents locally through an installed TeX distribution. The initial Windows implementation shall use MiKTeX and `latexmk`, with support for pdfLaTeX, XeLaTeX, LuaLaTeX, BibTeX, Biber, MakeIndex, and SyncTeX where available.

This document is intended to guide Codex and human contributors. It is deliberately divided into small, testable sprints. The implementation must not be attempted as one large change.

---

## 2. Product vision

TeXPulse Studio shall let a user:

1. Open or create a local LaTeX project.
2. Edit multiple source files in a modern code editor.
3. Save changes safely.
4. compile the project manually or automatically after edits.
5. See the latest valid PDF beside the source.
6. View compiler output, errors, warnings, and affected source lines.
7. Navigate from source to PDF and from PDF to source using SyncTeX.
8. Continue working without an internet connection.
9. Retain ownership of all project files on the local computer.
10. Recover safely from compiler failures, application crashes, and invalid documents.

The product should feel responsive like an online editor, but all compilation and file storage in the initial release shall remain local.

---

## 3. Goals

### 3.1 Primary goals

- Provide a dependable offline LaTeX editor for Windows.
- Compile standard LaTeX projects using the user's local MiKTeX installation.
- Provide fast feedback through debounced automatic compilation.
- Preserve the previous successful PDF when a new compilation fails.
- Present useful, structured diagnostics rather than only a raw log.
- isolate privileged filesystem and process operations from the renderer.
- Develop the product through small sprints with objective test evidence.
- Keep the architecture modular enough to add TeX Live, Tectonic, collaboration, and other platforms later.

### 3.2 Secondary goals

- Provide project templates.
- Provide recent-project management.
- Support ZIP import and export.
- Support configurable compiler recipes.
- Provide optional Git-aware project information.
- Support local revision snapshots in a later phase.

### 3.3 Non-goals for the first stable release

The first stable release shall not require or include:

- Cloud accounts.
- A hosted backend.
- Real-time multi-user collaboration.
- Chat or comments.
- Cloud project storage.
- Public project sharing.
- Full Overleaf compatibility.
- Automatic execution of arbitrary shell commands.
- A general-purpose terminal embedded in the application.
- Automatic installation of untrusted packages without user visibility.
- Mobile support.

These may be considered only after the offline single-user product is stable.

## 4. Definitions

| Term | Definition |
|---|---|
| Root file | The main `.tex` file passed to the LaTeX build tool. |
| Build | One attempt to turn the current project state into output files. |
| Build generation | A monotonically increasing identifier assigned to each requested build. |
| Last successful PDF | The newest PDF produced by a successful, non-stale build. |
| Stale build | A build whose generation is older than the newest requested generation. |
| Recipe | A named compiler configuration such as pdfLaTeX, XeLaTeX, or LuaLaTeX through `latexmk`. |
| Diagnostic | A structured error, warning, or informational compiler result linked to a file and line when possible. |
| SyncTeX | Source-to-PDF and PDF-to-source position mapping generated by TeX tooling. |
| Project metadata | TeXPulse-specific settings stored separately from source content. |
| Renderer | The Electron process that displays the user interface. |
| Main process | The privileged Electron process that accesses files and launches compilers. |
| Preload bridge | A restricted interface exposing approved main-process operations to the renderer. |

## 5. Stakeholders and user classes

### 5.1 Primary user

A student, researcher, lecturer, engineer, or technical writer who works with LaTeX locally and wants immediate PDF feedback without using a cloud editor.

### 5.2 Maintainer

A developer who maintains the Electron application, compiler adapters, tests, packaging, and documentation.

### 5.3 Future collaborator

## 6. Operating assumptions and constraints

1. The initial platform is Windows 11 x64.
2. MiKTeX is the initial TeX distribution.
3. The user may have MiKTeX installed outside its default path.
4. Executables must be discovered through `PATH`, known installation locations, or a user-selected path.
5. `latexmk` is the preferred build orchestrator.
6. The application must not depend on an internet connection after installation, except when the user intentionally asks MiKTeX to obtain missing packages.
7. The application must support paths containing spaces and non-ASCII characters.
8. The renderer must not receive unrestricted Node.js or filesystem access.
9. Shell escape must be disabled by default.
10. A real compiler may not be available in every automated test environment; therefore compiler adapters must support a deterministic fake compiler for tests.
11. The project should use TypeScript throughout the application code.
12. Package versions must be pinned through a lockfile.
13. The implementation must not mix WSL and Windows process paths in the MVP. A native Windows build should use native Windows Node.js, Electron, MiKTeX, and filesystem paths consistently.
14. All source projects remain user-owned local directories. TeXPulse must not silently rewrite user source files.

## 8. Architecture principles

### 8.1 Process separation

The Electron renderer shall be treated as untrusted.

- `nodeIntegration` shall be disabled.
- `contextIsolation` shall be enabled.
- The renderer shall access privileged operations only through a typed preload API.
- IPC channels shall be explicitly enumerated.
- Every IPC input shall be validated.
- File and process operations shall execute in the main process or a dedicated worker.

### 8.2 Modular compiler layer

The UI shall not call `latexmk` directly. It shall call a compiler service through an interface.

Suggested interface:

```ts
interface CompilerAdapter {
  probe(): Promise<ToolchainStatus>;
  compile(request: CompileRequest): Promise<CompileResult>;
  cancel(buildId: string): Promise<void>;
}
```

The first real adapter shall target MiKTeX plus `latexmk`. Tests shall use a fake adapter. Future adapters may target TeX Live or Tectonic.

### 8.3 Build state machine

The build controller shall implement explicit states:

```text
idle
→ debouncing
→ saving
→ queued
→ compiling
→ succeeded | failed | cancelled
→ idle
```

Each requested build shall receive a generation number. Results from stale generations shall never replace newer state.

### 8.4 Local-first data ownership

- Project source remains in the selected project directory.
- Generated files shall be placed in a configurable build directory, initially `.texpulse/build`.
- Application-level settings shall be stored in the operating system's application-data directory.
- Project-specific settings shall be stored in `.texpulse/project.json`.
- TeXPulse-specific files shall be suitable for inclusion in `.gitignore`.
- No source file shall be uploaded.

### 8.5 Testability

Pure logic such as tool detection, path validation, build queuing, root-file detection, and log parsing shall be separated from Electron APIs so it can be unit tested without launching Electron.

## 10. Functional requirements

### 10.1 Application startup and environment detection

- **FR-ENV-001:** The application shall detect whether `latexmk` is available.
- **FR-ENV-002:** The application shall detect available engines including pdfLaTeX, XeLaTeX, and LuaLaTeX.
- **FR-ENV-003:** The application shall detect BibTeX, Biber, MakeIndex, and SyncTeX independently.
- **FR-ENV-004:** The application shall display tool paths and version information.
- **FR-ENV-005:** The application shall let the user select a custom executable directory.
- **FR-ENV-006:** The application shall run a minimal toolchain self-test without modifying user projects.
- **FR-ENV-007:** The application shall explain missing prerequisites using actionable messages.
- **FR-ENV-008:** The application shall never report the toolchain as ready unless a real compile self-test succeeds or the user explicitly skips the test.

### 10.2 Project management

- **FR-PROJ-001:** The user shall be able to open an existing folder as a project.
- **FR-PROJ-002:** The user shall be able to create a new project from a minimal template.
- **FR-PROJ-003:** The application shall display a hierarchical project file tree.
- **FR-PROJ-004:** The user shall be able to create, rename, move, and delete files and folders after confirmation where data loss is possible.
- **FR-PROJ-005:** The application shall identify likely root files.
- **FR-PROJ-006:** The user shall be able to override the selected root file.
- **FR-PROJ-007:** The application shall remember recent projects.
- **FR-PROJ-008:** The application shall detect files changed by external programs.
- **FR-PROJ-009:** The application shall not follow filesystem links outside the project without an explicit policy decision.
- **FR-PROJ-010:** The application shall support paths containing spaces and Unicode characters.
- **FR-PROJ-011:** The application shall warn before overwriting externally modified unsaved content.
- **FR-PROJ-012:** The application shall allow ZIP export without including build artifacts by default.

### 10.3 Editing

- **FR-EDIT-001:** The application shall edit `.tex`, `.bib`, `.sty`, `.cls`, and plain-text project files.
- **FR-EDIT-002:** The editor shall provide LaTeX-aware syntax highlighting.
- **FR-EDIT-003:** The editor shall support undo and redo.
- **FR-EDIT-004:** The editor shall support find and replace.
- **FR-EDIT-005:** The editor shall show unsaved state.
- **FR-EDIT-006:** The user shall be able to save one file or all modified files.
- **FR-EDIT-007:** The editor shall preserve cursor and scroll state when switching files.
- **FR-EDIT-008:** The editor shall show diagnostic markers.
- **FR-EDIT-009:** The editor shall support configurable font size and word wrapping.
- **FR-EDIT-010:** The editor shall remain responsive while compilation runs.

### 10.4 Saving and file watching

- **FR-SAVE-001:** Autosave shall be configurable and enabled by default.
- **FR-SAVE-002:** Autosave shall use atomic replacement where technically practical.
- **FR-SAVE-003:** Save failure shall be visible and shall prevent a misleading compile request.
- **FR-SAVE-004:** External file changes shall be detected.
- **FR-SAVE-005:** External changes shall not silently destroy unsaved editor content.
- **FR-SAVE-006:** Temporary and build files shall not trigger infinite rebuild loops.

### 10.5 Compilation

- **FR-BUILD-001:** The user shall be able to trigger a manual build.
- **FR-BUILD-002:** The application shall support automatic builds after a configurable debounce period.
- **FR-BUILD-003:** The default automatic-build debounce shall be 800 milliseconds.
- **FR-BUILD-004:** Only one real compile process per project shall run at a time.
- **FR-BUILD-005:** New build requests during compilation shall replace intermediate queued requests with the newest requested state.
- **FR-BUILD-006:** Each build shall have a unique ID and generation number.
- **FR-BUILD-007:** Stale results shall not update the preview or current diagnostics.
- **FR-BUILD-008:** The user shall be able to cancel a running build.
- **FR-BUILD-009:** The build controller shall enforce a configurable timeout.
- **FR-BUILD-010:** The application shall capture standard output, standard error, exit code, start time, end time, duration, and invoked executable.
- **FR-BUILD-011:** Process arguments shall be passed without unsafe shell string concatenation.
- **FR-BUILD-012:** The compiler working directory shall be the validated project directory.
- **FR-BUILD-013:** Build outputs shall be separated from source files by default.
- **FR-BUILD-014:** The application shall support pdfLaTeX, XeLaTeX, and LuaLaTeX recipes through `latexmk`.
- **FR-BUILD-015:** The application shall respect `.latexmkrc` only under a clearly documented trust policy.
- **FR-BUILD-016:** The application shall retain the previous successful PDF when a new build fails.
- **FR-BUILD-017:** The application shall show whether the visible PDF is current or from the last successful build.
- **FR-BUILD-018:** The application shall provide a clean-build action.
- **FR-BUILD-019:** The application shall provide a safe auxiliary-file cleanup action.
- **FR-BUILD-020:** Missing-package output shall be recognized and explained.

### 10.6 PDF preview

- **FR-PDF-001:** A successful build shall display its PDF through PDF.js.
- **FR-PDF-002:** The viewer shall support page navigation, scrolling, zoom, fit-width, and fit-page.
- **FR-PDF-003:** Reloading a PDF shall preserve page, zoom, and approximate scroll position.
- **FR-PDF-004:** The viewer shall not load a PDF until the build process has completed and the output is readable.
- **FR-PDF-005:** The application shall handle file locking and partially written files without showing a corrupted preview.
- **FR-PDF-006:** The user shall be able to open the PDF in the system viewer.
- **FR-PDF-007:** The user shall be able to locate the generated PDF in the filesystem.
- **FR-PDF-008:** A stale build shall never replace the current preview.

### 10.7 Diagnostics and logs

- **FR-DIAG-001:** The application shall preserve the full raw build log.
- **FR-DIAG-002:** The application shall parse common LaTeX errors and warnings into structured diagnostics.
- **FR-DIAG-003:** Diagnostics should include severity, message, file, line, column when available, source tool, and raw excerpt.
- **FR-DIAG-004:** Selecting a diagnostic shall open the correct file and source line.
- **FR-DIAG-005:** The parser shall recognize at minimum undefined control sequences, missing files/packages, undefined references, undefined citations, overfull boxes, underfull boxes, emergency stops, Biber/BibTeX failures, and timeout/cancellation events.
- **FR-DIAG-006:** Parser failure shall not hide the raw compiler log.
- **FR-DIAG-007:** Diagnostics from stale builds shall not replace diagnostics from newer builds.
- **FR-DIAG-008:** The UI shall visually separate errors, warnings, and informational messages.

### 10.8 SyncTeX navigation

- **FR-SYNC-001:** Builds shall request SyncTeX output when supported.
- **FR-SYNC-002:** The application shall provide forward search from source position to PDF position.
- **FR-SYNC-003:** The application shall provide inverse search from PDF position to source file and line.
- **FR-SYNC-004:** SyncTeX commands shall use validated paths and argument arrays.
- **FR-SYNC-005:** The application shall show a clear non-fatal message when SyncTeX data is unavailable.
- **FR-SYNC-006:** SyncTeX results shall be tested with multi-file documents.

### 10.9 Settings

- **FR-SET-001:** The application shall store global settings locally.
- **FR-SET-002:** The application shall store project settings separately from global settings.
- **FR-SET-003:** Configurable settings shall include tool paths, selected recipe, root file, autosave, auto-build, debounce duration, timeout, build directory, editor font size, and PDF zoom mode.
- **FR-SET-004:** Settings shall be schema validated.
- **FR-SET-005:** Invalid or outdated settings shall fall back safely with a visible notice.
- **FR-SET-006:** The application shall support settings schema migration.

### 10.10 Recovery and resilience

- **FR-REC-001:** The application shall recover unsaved editor buffers after an abnormal shutdown where feasible.
- **FR-REC-002:** Recovery data shall not overwrite project files automatically.
- **FR-REC-003:** The user shall review and accept recovered content.
- **FR-REC-004:** A compiler crash shall not crash the application.
- **FR-REC-005:** A PDF rendering failure shall not destroy the last successful output.
- **FR-REC-006:** The application shall record a local diagnostic log suitable for troubleshooting.
- **FR-REC-007:** Logs shall avoid storing full document content by default.
- **FR-REC-008:** The user shall be able to clear recovery data and application logs.

### 10.11 Packaging and onboarding

- **FR-PACK-001:** The application shall produce a Windows installer and a development build.
- **FR-PACK-002:** The installer shall not silently bundle MiKTeX unless a later licensing and distribution decision explicitly allows it.
- **FR-PACK-003:** First launch shall run a toolchain setup wizard.
- **FR-PACK-004:** The setup wizard shall distinguish application readiness from TeX toolchain readiness.
- **FR-PACK-005:** The application shall provide a minimal sample project.
- **FR-PACK-006:** The release shall include user setup, troubleshooting, architecture, test, and contributor documentation.
- **FR-PACK-007:** Application updates shall be a later, separately reviewed feature and shall not block the first release.

### 10.12 Optional later features

The following requirements are deferred:

- Local revision history.
- Git status and commit assistance.
- Project templates marketplace.
- TeX Live adapter.
- Tectonic adapter.
- Linux and macOS packaging.
- Yjs collaborative documents.
- WebSocket or WebRTC collaboration.
- Comments and tracked changes.
- Hosted build workers.

Deferred features must not distort the MVP architecture or delay the stable offline release.

## 11. Non-functional requirements

### 11.1 Performance

- **NFR-PERF-001:** Typing shall not wait for file I/O or compiler work.
- **NFR-PERF-002:** Editor input handling should normally remain below 50 ms on the reference development machine.
- **NFR-PERF-003:** The build controller shall collapse multiple rapid requests into one newest queued build.
- **NFR-PERF-004:** A minimal reference document should refresh within three seconds after the debounce period on a correctly configured development machine.
- **NFR-PERF-005:** Opening a project containing 1,000 ordinary files shall not freeze the UI.
- **NFR-PERF-006:** Large generated folders shall be ignored by default.
- **NFR-PERF-007:** PDF reload shall preserve viewer state and avoid visible blanking where practical.

### 11.2 Reliability

- **NFR-REL-001:** A failed build shall never delete the previous successful PDF.
- **NFR-REL-002:** A stale build shall never become the displayed current result.
- **NFR-REL-003:** The application shall survive compiler exit codes, malformed logs, missing outputs, timeouts, and cancellations.
- **NFR-REL-004:** File writes shall minimize the chance of partial-file corruption.
- **NFR-REL-005:** Core state machines shall be deterministic and unit tested.

### 11.3 Security

- **NFR-SEC-001:** Electron security recommendations shall be followed.
- **NFR-SEC-002:** `nodeIntegration` shall be disabled in the renderer.
- **NFR-SEC-003:** `contextIsolation` shall be enabled.
- **NFR-SEC-004:** The preload API shall expose the minimum necessary capabilities.
- **NFR-SEC-005:** All IPC inputs shall be validated.
- **NFR-SEC-006:** User-controlled paths shall be canonicalized and checked.
- **NFR-SEC-007:** Shell escape shall be disabled by default.
- **NFR-SEC-008:** The compiler shall have a timeout.
- **NFR-SEC-009:** Log and output sizes shall be bounded.
- **NFR-SEC-010:** URLs opened externally shall be validated.
- **NFR-SEC-011:** Content Security Policy shall be configured.
- **NFR-SEC-012:** Dependency auditing shall be part of release gates.
- **NFR-SEC-013:** Multi-user compilation shall not be added without a sandboxed execution design and threat model.

### 11.4 Privacy

- **NFR-PRIV-001:** Source content shall remain local by default.
- **NFR-PRIV-002:** The application shall not include analytics in the initial release.
- **NFR-PRIV-003:** Crash or diagnostic uploads shall be opt-in if introduced later.
- **NFR-PRIV-004:** Logs shall redact user-home paths when exported where practical.

### 11.5 Compatibility

- **NFR-COMP-001:** The primary supported environment shall be Windows 11 x64.
- **NFR-COMP-002:** MiKTeX shall be the supported initial distribution.
- **NFR-COMP-003:** Projects with spaces and Unicode characters in paths shall be included in tests.
- **NFR-COMP-004:** The architecture shall not assume MiKTeX-specific behavior outside the MiKTeX adapter.
- **NFR-COMP-005:** TeX Live support shall be possible through another adapter.

### 11.6 Maintainability

- **NFR-MAINT-001:** TypeScript strict mode shall be enabled.
- **NFR-MAINT-002:** Public module contracts shall be typed and documented.
- **NFR-MAINT-003:** Compiler, filesystem, diagnostics, settings, and SyncTeX logic shall have independent modules.
- **NFR-MAINT-004:** Architectural decisions shall be recorded as ADRs.
- **NFR-MAINT-005:** Tests shall avoid arbitrary sleeps and nondeterministic timing.
- **NFR-MAINT-006:** The repository shall contain an `AGENTS.md` file with required commands and quality gates.

### 11.7 Accessibility and usability

- **NFR-UX-001:** Core actions shall be keyboard accessible.
- **NFR-UX-002:** Build states shall not be communicated by color alone.
- **NFR-UX-003:** Focus order shall be logical.
- **NFR-UX-004:** UI controls shall have accessible names.
- **NFR-UX-005:** Destructive file actions shall require confirmation.
- **NFR-UX-006:** Error messages shall say what failed, why it likely failed, and what the user can do next.

## 13. Compiler command policy

The initial pdfLaTeX recipe should be equivalent to:

```text
latexmk
-pdf
-synctex=1
-interaction=nonstopmode
-file-line-error
-halt-on-error
-outdir=<validated-build-directory>
<validated-root-file>
```

Requirements:

1. Use `spawn(executable, args, options)` or an equivalent argument-array API.
2. Do not use `shell: true` for ordinary builds.
3. Do not concatenate untrusted paths into a command string.
4. Validate the root file and build directory.
5. Record the resolved executable and arguments for diagnostics.
6. Redact sensitive path information when exporting support logs.
7. Treat `.latexmkrc` and shell escape as trust-sensitive features.

## 14. Testing strategy

### 14.1 Test levels

#### Unit tests

Required for:

- Toolchain path discovery.
- Version-output parsing.
- Root-file detection.
- Path canonicalization and project-boundary checks.
- Settings validation and migration.
- Build state transitions.
- Build generation and stale-result rejection.
- Debounce behavior using fake timers.
- Queue replacement.
- Timeout and cancellation logic.
- LaTeX log parsing.
- SyncTeX output parsing.
- Recovery-record logic.

#### Component tests

Required for:

- Editor status and unsaved indicators.
- Build-state indicator.
- Problems panel grouping and navigation.
- Settings validation messages.
- Toolchain setup wizard.
- PDF-viewer state restoration logic with mocked PDF loading.

#### Integration tests

Required for:

- Renderer-to-preload-to-main IPC contracts.
- Project open, read, edit, save, and external-change handling.
- Build controller with fake compiler process.
- Build controller with a small real MiKTeX fixture when available.
- Compilation result to diagnostics to editor navigation.
- Compilation result to PDF preview.
- Cancellation and timeout process cleanup.
- SyncTeX query service.

#### End-to-end tests

Required for critical user journeys:

1. Create a project, edit, compile, and view PDF.
2. Open an existing multi-file project and compile it.
3. Introduce an error, see a diagnostic, fix it, and recover the preview.
4. Trigger rapid edits and verify only the newest result is displayed.
5. Cancel a long build.
6. Restart after unsaved changes and review recovery.
7. Configure a custom MiKTeX path.
8. Use forward and inverse SyncTeX navigation.
9. Package and launch the installed Windows application.

#### Manual exploratory testing

Required for:

- Visual PDF quality.
- Resizing split panes.
- Keyboard navigation.
- Large projects.
- External editor conflicts.
- MiKTeX missing-package prompts.
- Antivirus and Windows file-lock behavior.
- High-DPI displays.
- Non-ASCII project paths.

#### Security testing

Required for:

- IPC input validation.
- Project path traversal attempts.
- Symlink or junction behavior.
- Malicious filenames.
- External URL validation.
- Shell metacharacters in paths.
- TeX shell-escape attempts.
- runaway compilation timeout.
- Oversized log and output handling.
- Dependency audit.

#### Performance testing

Required for:

- Rapid typing and debounce behavior.
- 1,000-file project-tree load.
- Repeated PDF refresh.
- Large compiler logs.
- Memory growth over repeated builds.
- Cancellation latency.

### 14.3 Coverage policy

- Core pure modules should target at least 85% statement and branch coverage.
- Coverage is not a substitute for meaningful assertions.
- Electron UI and packaging quality shall be judged through integration, E2E, screenshot, and manual checks in addition to coverage.
- Any untested safety-critical branch must be documented before a sprint can be accepted.

### 14.4 Quality gate after every sprint

A sprint is not complete until all applicable checks pass:

```text
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test:unit
pnpm test:integration
pnpm test:e2e
pnpm build
```

Not every early sprint must have UI E2E tests, but the command must either run relevant tests or clearly report that no E2E surface exists yet. Skipping a relevant failed test is not acceptable.

Every sprint report shall include:

- Implemented requirements.
- Changed files.
- Commands run.
- Test counts and results.
- Screenshots or generated PDF evidence when relevant.
- Known limitations.
- Security implications.
- Technical debt introduced.
- Exact next sprint.
- Git commit hash or proposed commit message.

## 15. Definition of Done

A backlog item is done only when:

1. The implementation satisfies its acceptance criteria.
2. Types are strict and no unexplained type suppression was added.
3. Relevant unit tests exist.
4. Relevant integration or E2E tests exist.
5. All quality-gate commands pass.
6. Failure paths are tested.
7. User-visible errors are actionable.
8. Documentation is updated.
9. No unrelated refactor is mixed into the change.

## 18. Acceptance scenarios

### AS-001: Minimal successful document

Given MiKTeX is configured and a minimal project is open, when the user edits the title and stops typing, the application saves the source, performs one newest build, and displays the updated PDF.

### AS-002: Failed build with usable previous preview

Given a successful PDF exists, when the user introduces an undefined command, the new build fails, the previous PDF remains visible, the UI marks it as the last successful output, and the error links to the source line.

### AS-003: Rapid edits

Given automatic compilation is enabled, when the user makes many edits quickly, the application does not run overlapping build processes and only the newest requested source state can update the preview.

### AS-004: Build cancellation

Given a build is running, when the user cancels it, the compiler process tree is terminated, the UI enters the cancelled state, and no later output from that build updates the preview.

### AS-005: Missing toolchain

Given MiKTeX or `latexmk` is unavailable, when the application starts, it reports which executable is missing, does not claim readiness, and provides setup instructions without crashing.

### AS-006: Multi-file diagnostic

Given `main.tex` includes `chapters/intro.tex`, when the included file contains an error, the Problems panel links to the correct included file and line.

### AS-007: SyncTeX

Given a successful build with SyncTeX data, when the user requests forward search, the PDF viewer moves to the corresponding location; when the user invokes inverse search in the PDF, the editor opens the matching file and line.

### AS-008: Recovery

Given unsaved content exists and the application terminates abnormally, when it starts again, the user is offered recoverable content and the project file is not overwritten until the user confirms.

### AS-009: Project boundary

Given a malicious or malformed renderer request asks to read outside the open project, the main process rejects the request and records a security-relevant diagnostic without exposing the external file.

### AS-010: Packaged beta

Given a clean supported Windows account and a configured MiKTeX installation, when the user installs TeXPulse Studio, opens the sample project, edits it, and compiles, the PDF preview works without development tooling.

## 19. Risks and mitigations

| Risk | Mitigation |
|---|---|
| MiKTeX path and package behavior vary | Toolchain probe, custom path, self-test, clear diagnostics |
| TeX can run for an excessive time | Timeout, cancellation, bounded output, shell escape off |
| Electron renderer compromise | Context isolation, no Node integration, narrow validated IPC |
| PDF file locks or partial writes | Load only after process completion; retry completed file reads safely |
| Stale build race | Generation IDs and newest-result checks |
| File-watcher loops | Ignore build directories and self-generated events |
| Log formats vary | Golden fixtures, tolerant parser, always preserve raw log |
| WSL/Windows path confusion | Native Windows MVP and explicit environment ADR |
| Codex attempts too much at once | One-sprint execution gate and required sprint report |
| Tests pass only with mocks | Conditional real MiKTeX smoke suite and packaged E2E test |
| Collaboration introduces major security complexity | Defer it; require separate SRS and threat model |
